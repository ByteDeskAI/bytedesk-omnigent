# ADR: Omnigent self-documenting capability contract

**Status:** Proposed (2026-06-26)  
**Scope:** `bytedesk-omnigent` API/schema/docs/SDK capability metadata. No ByteDesk Platform service changes in this proposal.

## Context

BDP-2512 asks for a design discussion and ADR proposal for a self-documenting
Omnigent application after the BDP-2503 extension/runtime-flag work lands. The
implementation should be planned as follow-up work, not hidden inside this
ticket.

Omnigent already has the important primitives:

- FastAPI exposes the live HTTP spec through `/openapi.json`, Swagger UI at
  `/docs`, and ReDoc at `/redoc`.
- `scripts/dump_openapi.py` builds the app, post-processes the generated spec to
  OpenAPI 3.2, injects the `ServerStreamEvent` union, and writes the checked-in
  root `openapi.json`.
- `tests/server/test_openapi_drift.py` fails when `openapi.json` differs from
  the generated app spec.
- `GET /v1/schema/events` returns the standalone ServerStreamEvent JSON Schema,
  and `GET /v1/_capabilities` advertises the event-schema version, hash, and
  variant count.
- `omnigent.kernel.pluggable.manifest.capability_manifest()` projects live seam
  registries from the central `SEAMS` table into `_capabilities`.
- `omnigent.sdk` decorators preserve metadata for tools, policies, harnesses,
  background tasks, routers, interceptors, and service providers while compiling
  into the same `OmnigentExtension` protocol hooks as hand-written extensions.

The current gap is not the absence of OpenAPI generation. The gap is that HTTP,
event schemas, pluggable seams, SDK-decorated extension contributions, config
descriptors, tools, policies, harnesses, and background capabilities are not yet
materialized as one reviewable, drift-gated capability contract.

## Decision

Keep the root `openapi.json` generated by `scripts/dump_openapi.py` as the
checked-in canonical HTTP/event contract. The FastAPI application remains the
runtime source for HTTP routes; CI and local tests compare the generated output
to the committed artifact. Production pods must not write generated files back
to the repository or mutate the committed contract at boot.

Add a separate boot-time capability snapshot in a follow-up implementation. At
server startup, after first-party and discovered extensions are installed, the
server should compute an immutable in-process snapshot that includes:

- OpenAPI version, spec path, and content hash.
- Event-schema version, content hash, and variant count.
- `capability_manifest()` seam entries, including registered names, active
  implementation, default implementation, override environment variable, and
  any manifest errors.
- SDK-decorated extension contributions for tools, policies, harnesses, routers,
  tool interceptors, background tasks, and config descriptors.
- Contribution provenance: extension name, Python module/object when safe to
  expose, protocol hook or SDK decorator, lifecycle phase, auth/admin
  requirement for exposed routes, and redacted configuration metadata.

Expose that snapshot through a stable read-only schema surface. The likely shape
is a new `GET /v1/schema/capabilities` endpoint, plus an additive hash/version
summary inside `GET /v1/_capabilities` so clients can detect drift without
fetching the full document.

Materialize reviewable artifacts only in local development and CI. A follow-up
script should generate a checked-in `capabilities.json` snapshot from the same
builder used at boot, with a drift test mirroring `test_openapi_drift.py`.
Runtime boot computes and serves the snapshot; it does not commit or rewrite
artifacts.

Generate human documentation from the same snapshot rather than from
hand-maintained capability tables. Use:

- FastAPI/OpenAPI plus ReDoc or Swagger UI for HTTP route documentation.
- Pydantic v2 JSON Schema for typed request, response, event, and capability
  payloads.
- MkDocs plus mkdocstrings for narrative docs and Python API reference, because
  the repo is Python-first and the docs need to combine conceptual pages with
  generated API/catalog pages.
- Spectral or Redocly linting for OpenAPI quality gates once the contract grows
  beyond the current drift test.
- Schemathesis later for behavioral/property tests against the generated
  OpenAPI contract, after the static contract is complete.

Do not introduce apispec or connexion as the primary generator. FastAPI already
owns the route/schema graph, so a parallel spec-first stack would create a
second source of truth. Do not use pdoc or Sphinx autoapi as the first
documentation engine for the capability catalog; they are useful API-reference
tools but weaker for combining product documentation, generated OpenAPI pages,
and extension capability catalogs.

## Completeness Invariants

- Extension routers must be installed before OpenAPI generation so first-party
  and discovered routes appear in `openapi.json`.
- Any new SSE endpoint must be added to `scripts/dump_openapi.py`'s SSE route
  inventory and must use the shared event-schema builder.
- The boot-time snapshot and the checked-in `capabilities.json` artifact must be
  produced by the same builder.
- SDK-authored extensions must expose enough metadata through decorators or
  config descriptors for docs generation; hand-written extensions should have a
  lightweight metadata adapter rather than bespoke markdown tables.
- Snapshot generation must redact secrets and distinguish active, disabled, and
  unavailable capabilities instead of omitting unavailable entries.

## Consequences

- Omnigent gets one self-documenting contract that covers HTTP, events, seams,
  SDK extension contributions, and operational configuration.
- SDK/client generation can continue to consume `openapi.json` and
  `/v1/schema/events` while richer admin/docs tooling consumes the capability
  snapshot.
- Extension authors must provide accurate metadata when they add tools, routes,
  policies, harnesses, config, or background work.
- CI gains one more drift gate when `capabilities.json` is introduced.
- Boot work must stay cheap: compute once after extension installation, cache the
  result, and avoid network calls or provider factory side effects while
  describing capabilities.

## Risks

- Conditional extension loading can produce different contracts across
  environments. The snapshot must show both active state and provenance so
  operators can tell whether a capability is absent, disabled, or unavailable.
- Over-eager introspection can leak implementation details or secret names.
  Snapshot payloads must whitelist fields and redact configuration values.
- A docs generator that imports too much runtime code could slow boot or break
  the runner hot path. Snapshot generation belongs in the server startup path,
  after extension installation, and must not be imported by runner-only code.

## Non-goals

- Implementing the capability snapshot generator in BDP-2512.
- Replacing FastAPI's OpenAPI generation.
- Generating SDK clients from the new capability snapshot.
- Adding a web administration UI for the generated docs.
- Exposing secrets, raw environment values, or provider credentials.
